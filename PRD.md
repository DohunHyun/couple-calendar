# PRD — Couple Calendar

> 제품 요구사항 문서. **기획 의도(target)가 정본**이며, 각 항목의 **현재 구현 상태**를 함께 표기한다.
> 이전 버전(v1.0)은 "현재 코드가 하는 것"을 의도처럼 기록한 오류가 있었다. v2.0은 **나현 기획서 + 12대 명세를 정본으로** 두고, 실제 코드 정독으로 확인한 상태를 병기한다.
> 기술 상세는 [spec.md](spec.md) 참고.

- **문서 버전**: 3.0 (기기 캘린더 연동을 메인 기능으로 추가)
- **최종 수정**: 2026-06-28
- **상태 범례**: ✅ 충족 / ⚠️ 부분 / ❌ 미충족 / 🐛 버그 / 🔜 계획(설계 확정·미구현)

## 1. 제품 개요

연인 두 사람이 하나의 그룹으로 묶여 **일정을 함께 보면서도, 각자의 사생활은 지키는** 커플 캘린더 앱.
**각자 평소 쓰던 폰 기본 캘린더를 그대로 쓰면**, 선택한 일정이 자동으로 커플 앱에 모인다(§3-A).

핵심 가치 네 가지:
1. **따로 입력 안 함** — 폰 기본 캘린더에 등록한 일정이 자동으로 커플 앱에 흘러 들어온다. (v3.0 메인)
2. **공유** — 둘이 함께하는 일정을 한 화면에서.
3. **사생활 보호** — 개인 일정은 작성자 본인에게만.
4. **관계 맥락** — 연애 시작일 기반 D-Day를 곁에.

사용자: 기획자 **나현(Galaxy S26)**, 개발자 **도훈(iPhone)** 같은 연애 중 커플. 모바일 앱(Capacitor, iOS/Android) 우선, 라이트 모드 미니멀 UI.

## 2. 문제 정의

일반 캘린더 공유는 **전부 공개**(사생활 없음) 또는 **완전 분리**(공유 일정 중복 입력) 양극단으로 흐른다. 커플에게 필요한 건 **"공유는 같이, 개인은 따로"가 한 캘린더 안에서 자연스럽게 나뉘는 것**.

특히 **"완전 분리 → 중복 입력"** 문제가 크다: 각자 폰 캘린더에 이미 일정을 적는데, 커플 앱에 또 적어야 하면 누구도 꾸준히 안 쓴다. 본 제품은 ① **폰 기본 캘린더를 자동 연동**해 재입력을 없애고(§3-A), ② 일정마다 `SHARED / PRIVATE`로 공유/사생활을 나눠 두 문제를 동시에 해결한다.

## 3-A. 메인 기능: 기기 캘린더 연동 (v3.0 신규 — 설계 확정·미구현)

### 한 줄 정의
각 사용자가 자기 폰 기본 캘린더에서 **선택한 캘린더**의 일정을 커플 서비스로 **자동 가져와**(폰 → 서비스, 단방향) 보여준다. 기본은 본인만(PRIVATE), 원하는 것만 공유(SHARED).

### 왜 메인인가
§2의 "완전 분리 → 중복 입력" 문제를 정면 해결. 사용자는 **평소 쓰던 폰 캘린더를 그대로 쓰고**, 그게 커플 앱으로 흘러 들어온다.

### 동작 (v1: 폰 → 서비스 단방향)
1. 앱이 캘린더 접근 권한 요청.
2. 사용자가 **연동할 기기 캘린더를 선택**(설정).
3. 앱 열 때 자동 + 당겨서 새로고침 → 선택 캘린더의 일정을 읽어 서비스로 업로드.
4. 상대는 SHARED로 표시된 일정만 본다.

### 확정 결정 (2026-06-28)
| 항목 | 결정 |
|------|------|
| 방향 | 폰 → 서비스 **단방향** (v1) |
| 범위 | 사용자가 **연동할 캘린더 선택** |
| 공유 기본값 | 설정에서 PRIVATE/SHARED 선택, **디폴트 PRIVATE**, 일정마다 개별 변경(sticky) |
| 인앱 일정 | LOCAL 인앱 생성 **유지** (폰 미러링 DEVICE와 공존) |
| 카테고리/색 | 연동한 **기기 캘린더별로 색 보존**(자동 카테고리 생성) |
| 구글 동기화 | 네이티브가 **대체** (구글 API 동기화 제거, 공휴일은 앱 자체 데이터로 유지) |

### 제약 (정직하게 명시)
- **"자동"의 한계**: 모바일은 앱이 꺼져 있을 때 캘린더를 실시간 감시하지 못한다. v1의 "자동" = **앱 열 때 + 당겨서 새로고침**. 실시간/백그라운드 동기화는 로드맵.
- DEVICE 일정은 앱 안에서 **공유 토글만** 가능(제목·시간 수정은 폰 캘린더에서). v1은 앱→폰 쓰기 없음.
- **삭제 전파**: 기기에서 지운 일정은 서비스에서 숨김(기기가 원본).
- 가져온 일정에 준 **공유 설정은 재동기화돼도 유지**(sticky override).

기술 설계는 [spec.md §8-A](spec.md) 참고.

## 4. 페르소나 & 핵심 시나리오

1. 도훈이 카카오/구글로 로그인 → **초대코드 생성** → 나현에게 공유.
2. 나현이 로그인 → **초대코드 입력** → 두 사람 자동 연결(SSE 실시간 전환).
3. 각자 닉네임·연애 시작일 설정(건너뛰기 가능).
4. 각자 설정에서 **연동할 폰 캘린더를 선택**(예: 나현은 "개인" 캘린더만). 이후 폰에 적은 일정이 앱에 자동으로 들어온다(기본 PRIVATE).
5. 메인 달력에서 날짜 탭 → 하단 시트로 그 날 일정.
6. "주말 데이트"는 **SHARED**로 바꿔 둘 다 보이게, "병원 예약"은 **PRIVATE** 그대로 본인만.

## 5. 사용자 플로우 (온보딩 단계)

서버 `onboardingStage`로 화면 분기: `로그인 → LINK → PROFILE → MAIN`.

- `LINK`: 커플 미연결. **연동 선택 화면**(생성 / 입력).
- `PROFILE`: 연결됨, 프로필 미완료.
- `MAIN`: 메인 달력.

화면 단계: 로그인(UI-ON-001) → 연동선택(UI-ON-002) → 코드생성(UI-ON-003, 소켓대기) / 코드입력(UI-ON-004, 검증) → 프로필(UI-ON-005).

## 6. 핵심 제품 규칙 (불변 비즈니스 룰)

구현 매핑은 [spec.md §5](spec.md).

- **R1. 1인 1커플** — 이미 커플인 사용자는 다른 코드로 연결 불가.
- **R2. 자가 연결 불가** — 자기 초대코드로는 join 불가.
- **R3. 연동 상태 분리** — 생성 시 생성자 couple_id는 NULL, join 성공 시 양쪽 동시 매핑.
- **R4. 가시성** — 일정/카테고리는 *본인 소유* 또는 *같은 커플의 SHARED* 만 보인다.
- **R5. 수정/삭제 권한** — 본인 소유 또는 같은 커플의 SHARED 일정만.
- **R6. 공유 삭제 확인** — 남이 등록한 SHARED 일정 삭제 시 "[작성자]님이 등록한 공유 일정입니다. 함께 삭제하시겠습니까?" 확인.
- **R7. 구글 비파괴** — 구글 일정 삭제는 `is_hidden` 마스킹, 구글 API DELETE 금지.
- **R8. 닉네임 ≤10자**.
- **R9. 색상 12색 파스텔 제한**.
- **R10. 메인 달력 순백 배경 유지**.
- **R11. 제목 ≤50자 / 메모 ≤200자**.
- **R12. 푸시 분기** — PRIVATE→본인, SHARED→커플 2인.
- **R13. 기기 동기화는 선택 캘린더만** — 사용자가 고른 기기 캘린더의 일정만 가져온다(사생활 보호).
- **R14. 동기화 일정 기본 PRIVATE** — 가져온 기기 일정의 기본 공개범위는 PRIVATE. 설정에서 기본값 변경 가능, 일정마다 개별 override(sticky).
- **R15. 기기가 원본(단방향)** — DEVICE 일정의 제목·시간은 기기에서만 수정. 앱은 공유 토글만. 기기 삭제 → 서비스 숨김.
- **R16. 기기 캘린더 색 보존** — 연동한 기기 캘린더별로 카테고리를 만들어 색을 유지한다.

## 8. 비범위 (Out of Scope, v3.0 기준)

- **3명 이상 그룹** — 1인 1커플 고정.
- **채팅/메신저**.
- **양방향/서비스→폰 동기화** — v1은 폰→서비스 단방향만. (양방향은 로드맵)
- **실시간/백그라운드 자동 동기화** — v1은 앱 열 때 + 새로고침.
- **앱에서 DEVICE 일정의 제목·시간 편집** — 폰 캘린더에서만(앱은 공유 토글만).
- **가입 없이 둘러보기(프리뷰)** — 의도상 제외(현재 G5로 잔존).
- ~~구글 외 외부 캘린더(Apple/Outlook)~~ → v3.0부터 **범위 안**: 기기 캘린더 연동은 폰 저장소를 읽으므로 iCloud·삼성·구글 등 폰에 들어온 모든 계정 일정을 포함.
